# EA-AOL Agent Contract

This repository is contract-first, schema-backed, and fail-closed. Any coding assistant working here must treat `contracts/` and `tests/` as the source of truth and `src/` as the implementation layer that follows them.

## Required Read Order

Before proposing or generating code, read these in order:

1. `contracts/system_contract.md`
2. `contracts/error_contract.md`
3. `contracts/operation_manifest.json`
4. The relevant file under `contracts/operations/`
5. The relevant request and response schemas under `contracts/schemas/`
6. The relevant tests under `tests/`

Do not start from `src/` and guess the contract after the fact.

## Repository Rules

- Treat this repository as an independent EA-AOL project, not as a mixed experiment space.
- Keep request and response handling fail-closed.
- Emit only contract-valid success or error envelopes from public runtime surfaces.
- Do not introduce warning-only paths, soft failures, or best-effort coercions.
- Do not add undocumented operations.
- Do not add runtime behavior that is not backed by contract docs, schemas, manifest entries, and tests.

## Required Change Protocol

When adding or changing an operation, update all of the following together:

1. `contracts/operations/<operation>.md`
2. `contracts/schemas/<operation>.schema.json`
3. `contracts/schemas/<operation>.response.schema.json`
4. `contracts/operation_manifest.json`
5. `src/entrypoint.py`
6. `tests/test_entrypoint_contract.py`

If a change does not need a new operation, update only the layers that are still contract-relevant. Do not modify `src/` alone when the effective contract changes.

## CI Expectations

Local validation entrypoints are:

- `python tools/ci_gate.py validate-schema`
- `python tools/ci_gate.py validate-completeness`
- `python tools/ci_gate.py validate-response-contract`
- `python tools/ci_gate.py validate-governance`
- `python tools/ci_gate.py unit-tests`

All gates are binary. The only valid outcomes are `PASS` or `FAIL`.

## Preferred Agent Behavior

- Start by identifying the contract boundary affected by the task.
- Make the smallest change that satisfies the contract.
- Preserve deterministic normalization behavior.
- Prefer extending existing canonicalization and validation paths over creating parallel logic.
- If a request is ambiguous, choose the stricter fail-closed interpretation.

## Forbidden Patterns

- Adding new fields without schema updates
- Returning ad hoc error shapes
- Introducing hidden retries or fallback behavior
- Mutating the manifest, schema set, or handler registry out of sync
- Treating README text as sufficient in place of contract files

## Definition of Done

A change is not complete until:

- contract documents are in sync
- manifest completeness still passes
- request and response shapes are validated
- tests cover the changed contract surface
- `tools/ci_gate.py` passes for all required gates
